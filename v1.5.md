# 今天吃什么 v1.5 项目学习记录

## 使用约定

- 后续每次执行新的更改计划前，先重读本文件与 `structure.md` 的“项目现状”“实现结构”“偏差与风险”部分，再开始设计和修改。
- 本文件偏重“项目做了什么、文档怎么描述、代码真实实现到哪一步”。
- `structure.md` 偏重“目录、文件职责、调用链、依赖关系”。

## 1. 项目一句话定义

`今天吃什么-(what-to-eat)1.5` 是一个基于 React + TypeScript + Vite 的前端饮食推荐应用。它以本地硬编码食谱库为基础，通过偏好加权、禁忌剔除、关键词搜索和轻量随机因子，生成早餐单品或午晚餐“三菜一汤”推荐。

## 2. 当前版本真实能力

### 已实现

- 首页支持三种入口：
  - 按时段直接推荐早餐、午餐、晚餐
  - 输入关键词搜索菜名/食材
  - 选择“我爱吃 / 忌口”偏好标签
- 早餐输出 1 道菜。
- 午餐和晚餐输出 3 道菜：
  - 1 道荤菜
  - 1 道素菜
  - 1 道汤
- 支持单独替换套餐中的某一道菜。
- 支持整桌重新生成。
- 支持查看菜品图片、描述、食材、步骤。
- 支持将推荐记录保存到浏览器 `localStorage`。
- 支持导出历史记录为 JSON。
- 支持 Web Share 或复制文字分享当前推荐菜单。
- 支持模糊搜索：
  - 菜名完全匹配
  - 菜名包含
  - 食材包含
  - Levenshtein 编辑距离近似匹配

### 未实现但文档中多次宣称

- 没有真正接入 Gemini 或其他 LLM。
- 没有联网生成图片，图片只是拼接 Bing 图片搜索 URL。
- 没有“餐厅推荐”实际流程，相关组件未接入。
- 没有食材库存管理。
- 没有营养成分计算。
- 没有时令、天气、身体状态、生理期等推荐逻辑。
- 没有历史记录页面，只能导出 JSON。
- 没有“社区分享”模块，只是文本分享。
- 没有夜宵模式。
- 没有 600+ 菜谱，当前食谱库共 93 道。

## 3. 开发文档阅读结论

### `项目企划分析书.md`

这是本项目最贴近产品策划层的说明文档，核心观点是：

- 用户痛点是“饮食决策疲劳”，不是单纯“缺菜谱”。
- 产品定位从“菜谱展示”升级为“智能饮食决策引擎”。
- 核心动作是：
  - 聚合菜谱
  - 结构化存储
  - 多维评分
  - 可视化推荐
- 文档中设定的算法框架为：
  - 搜索匹配
  - 偏好加权
  - 禁忌一票否决
  - 随机因子
- 文档中的理想架构比代码更大，包含营养均衡、偏好学习、规划能力。

我的结论：

- 这份文档准确描述了当前代码的“算法方向”，但明显高于代码实装程度。
- 它对未来愿景和比赛包装贡献很大，但不能直接当作当前代码事实。

### `2025_SHU_AI_Contest_Report.md`

这是一份比赛研究报告，主要作用是：

- 将项目包装为“智能饮食决策系统”。
- 强调交互风格、决策效率、食材利用率和实验效果。
- 解释模糊匹配、加权推荐、历史记录与分享闭环。
- 补充了图表、实验结论和未来展望。

我的结论：

- 它比企划书更“结果导向”，用于对外展示和答辩。
- 报告中关于 600+ 菜谱、完整历史闭环、图表实验等内容，和当前代码并不完全一致。
- 报告是“展示版本叙事”，不是“源码级真实状态说明”。

## 4. 代码真实产品形态

### 用户主流程

1. 打开首页。
2. 选择偏好标签，或直接点击早餐/午餐/晚餐。
3. 系统进入加载态。
4. 从本地食谱库挑选结果。
5. 将结果包装成带图片 URL 的展示数据。
6. 进入结果页。
7. 用户可：
   - 返回
   - 重新生成
   - 替换套餐中的单道菜
   - 分享
8. 推荐结果写入历史记录。

### 搜索流程

1. 用户输入关键词。
2. 系统遍历所有菜谱并打分。
3. 只保留有匹配分数的结果。
4. 按分数降序排列。
5. 用户点击某一条结果查看详情。

### 推荐逻辑真实规则

- 禁忌优先：
  - 如果菜名或任一食材命中 `dislikes`，该菜直接剔除。
- 搜索加分：
  - 菜名完全等于关键词：`+100`
  - 菜名包含关键词：`+50`
  - 食材包含关键词：`+30`
  - 菜名模糊相似：`+20 * 相似度`
  - 食材模糊相似：`+15 * 相似度`
  - 描述包含关键词：`+10`
- 喜好加分：
  - 菜名或食材命中一个 like：`+10`
- 随机因子：
  - 搜索模式：`0~2`
  - 非搜索模式：`0~5`
- 选择策略：
  - 先按分数排序
  - 再从前 20% 候选中随机挑 1 条

### 午晚餐组装逻辑

- 午晚餐固定返回三道：
  - `MEAT_RECIPES`
  - `VEG_RECIPES`
  - `SOUP_RECIPES`
- 素菜会尝试避开和荤菜重名或食材冲突过多。
- 食材冲突判断非常简单：
  - 两道菜有 2 个及以上完全相同食材字符串，就认为冲突。
- 汤不做冲突校验。

## 5. 数据层真实情况

### 食谱数量

- 早餐：33
- 荤菜：28
- 素菜：22
- 汤：10
- 总计：93

### 数据结构

每道菜统一为：

- `dishName`
- `description`
- `ingredients`
- `steps`

说明：

- 没有卡路里、蛋白质、烹饪时长等结构化字段。
- 文档里提到的营养属性，目前只存在于报告叙事和可视化脚本，不在业务数据里。

## 6. UI 与交互风格

当前界面风格很明确：

- 暖色系
- 大圆角
- 毛玻璃感
- 漂浮背景光斑
- 卡片式交互
- 强视觉首屏

实现方式并不标准工程化：

- 依赖 `index.html` 里注入的 Tailwind CDN
- 组件内部大量直接写 Tailwind 类名
- `index.css` 仍保留 Vite 默认样式

结论：

- 视觉语言已经形成，但底层样式体系并不干净。
- 当前 UI 是“能跑且好看”的原型化实现，不是已经整理好的设计系统。

## 7. 历史记录与分享真实情况

### 历史记录

- 写入位置：浏览器 `localStorage`
- 键名：`what_to_eat_history`
- 最多保存 50 条
- 每条记录包含：
  - id
  - timestamp
  - dateStr
  - mealType
  - dishes

### 分享

- 若浏览器支持 `navigator.share`，直接调用系统分享。
- 否则把菜单文字复制到剪贴板。
- 分享的是文字，不是图卡。

## 9. 关键偏差与风险

### 文档与代码偏差

- 文档多次提到 600+ 菜谱，实际只有 93 道。
- 文档写“Gemini/AI/智能推理”，代码是纯本地规则引擎。
- 文档写“营养分析”，代码无营养字段。
- 文档写“历史记录模块”，代码只有导出，没有浏览界面。
- 文档写“餐厅/地图/社交分享”，餐厅模块未接入，分享仅文本。

### 结构命名偏差

- `geminiService.ts` 命名会误导后续开发。
- `RestaurantDisplay.tsx` 与 `Restaurant` 类型是未接入资产。
- `AppState` 中的 `PREFERENCES` 实际未使用。

### 技术实现风险

- 使用 Tailwind CDN，而不是标准 npm/PostCSS/Tailwind 配置。
- `index.html` 同时保留 importmap，和 Vite/npm 模式混合，容易让人误判构建方式。
- 图片依赖外部 Bing 搜索 URL，稳定性不可控。
- 偏好只存在内存，不会持久化到浏览器。

## 10. 后续开发时必须记住的基线

- 这是一个“本地规则推荐应用”，不是一个真正联网 AI 系统。
- 当前最真实的业务核心只有三件事：
  - 按时段推荐
  - 按关键词搜索
  - 按偏好过滤与加权
- 任何后续改动都要先判断：
  - 是在扩展现有规则系统
  - 还是在把文档中的“未来愿景”真正落地
- 以后开始任何改动前，先回读本文件与 `structure.md`，避免被报告叙事误导。
